O que e o AIOX?

Antes de qualquer coisa, entenda os conceitos basicos.

⚙ Agentes de IA

O AIOX tem 12 agentes especializados. Cada um faz uma coisa — como membros de um time real. Voce chama eles com /AIOX:agents:nome e pede tarefas com *comando.

★ Comandos com *

Os comandos com asterisco sao acoes que voce pede para o agente. Exemplo: voce ativa o /AIOX:agents:dev e digita *develop — ele implementa o codigo. Cada agente tem seus proprios comandos.

📑 Stories

Uma story e a menor unidade de trabalho. Descreve uma funcionalidade do ponto de vista do usuario. Exemplo: "Como usuario, quero fazer login com Google."

📦 Epics

Um epic e um objetivo grande demais para uma story so. Exemplo: "Sistema de Autenticacao" e um epic com varias stories (login, registro, reset de senha...).

Referencia Rapida — Passo a Passo

O fluxo completo, do zero ao deploy. Clique em cada fase para expandir.

🛠

Fase 1 — Planejamento

Definir O QUE vai ser construido. 4 passos, 5 agentes.
1

Entender o projeto /AIOX:agents:analyst

Pesquisa de mercado, analise de concorrentes, brief completo. E a fundacao — sem isso, voce constroi a coisa errada.
*create-project-brief *create-project-brief --mode yolo *perform-market-research *create-competitor-analysis *brainstorm {topico}
2

Definir requisitos /AIOX:agents:pm

Transforma o brief em um PRD — o documento que lista TUDO que o produto precisa ter. Requisitos funcionais, nao-funcionais, restricoes.
*create-prd *create-prd --mode yolo *create-brownfield-prd *create-epic
3

Projetar arquitetura + experiencia /AIOX:agents:architect /AIOX:agents:ux-design-expert ⇄ paralelo

/AIOX:agents:architect decide COMO o sistema sera construido (tecnologias, APIs, banco).
/AIOX:agents:ux-design-expert define como o usuario vai INTERAGIR (fluxos, telas, design system).
Rodam ao mesmo tempo porque nao dependem um do outro.
*create-full-stack-architecture *create-full-stack-architecture --mode yolo *create-front-end-spec *create-front-end-spec --mode yolo *wireframe {low|mid|high} *create-backend-architecture
4

Quebrar PRD em partes /AIOX:agents:po

Fatia o PRD grande em pedacos menores (epics e stories). Sem isso, o Scrum Master nao consegue criar stories — e como dar um livro inteiro pra alguem sem dividir em capitulos.
*shard-doc {doc} {destino} *shard-doc {doc} {destino} --mode yolo *backlog-prioritize
💻

Fase 2 — Desenvolvimento

Implementar cada story. Repete para cada story do epic.
5

Criar a story /AIOX:agents:sm Draft

O Scrum Master cria a user story com criterios de aceitacao (o que define "pronto"), lista de tasks e notas tecnicas para o dev.
*draft *draft --mode yolo *story-checklist
6

Validar a story /AIOX:agents:po Ready

O Product Owner revisa a story com um checklist de 10 pontos. Se a nota for 7 ou mais, a story e aprovada e o dev pode comecar.
*validate-story-draft *validate-story-draft --mode yolo *stories-index
7

Implementar /AIOX:agents:dev InProgress

O dev pega a story aprovada e implementa: escreve codigo, cria testes, roda lint e build. Voce escolhe o modo (veja abaixo).
*develop *develop-yolo *develop-preflight *build *run-tests
8

Revisar qualidade /AIOX:agents:qa InReview

O QA revisa o codigo, verifica criterios de aceitacao, seguranca e qualidade. Resultado: PASS (ok) / CONCERNS (ok com notas) / FAIL (volta pro dev).
*review {story} *review {story} --mode yolo *gate {story} *code-review {escopo} *security-check {story}
9

Fechar a story /AIOX:agents:po Done

O Product Owner faz o aceite formal e marca a story como concluida. Pronto, essa feature existe!
*close-story *close-story --mode yolo
↺ Repita os passos 5 a 9 para cada story ate fechar o Epic inteiro
🚀

Fase 3 — Entrega

Publicar o codigo. Exclusivo do /AIOX:agents:devops.
10

Verificar e enviar /AIOX:agents:devops

Roda todos os quality checks e envia o codigo para o GitHub. Nenhum outro agente pode fazer push — so o /AIOX:agents:devops.
*pre-push *push *push --mode yolo
11

Criar Pull Request /AIOX:agents:devops

Cria o PR no GitHub para revisao do codigo antes de ir para produção.
*create-pr *create-pr --mode yolo *version-check
12

Release /AIOX:agents:devops OPCIONAL

Cria uma release versionada com changelog. Util quando o epic inteiro esta pronto e voce quer marcar uma versao.
*release *release --mode yolo *configure-ci *setup-github

Modos de Execucao

Todo comando no AIOX pode rodar em diferentes modos. O modo controla quanta autonomia o agente tem — quanto ele faz sozinho vs. quanto ele te pergunta.

PADRAO
Interativo

Voce participa

O agente faz perguntas ao longo do caminho. Voce toma as decisoes importantes juntos. E o modo padrao — se voce nao especificar nada, e esse que roda.
Perguntas5 — 10
Controle████▁ Alto
Melhor paraMaioria dos casos
*comando
YOLO

Ele resolve

O agente toma TODAS as decisoes sozinho, sem perguntar. Voce so ve o resultado final. Para ativar, adicione --mode yolo ao final do comando.
Perguntas0 — 1
Controle█▁▁▁▁ Minimo
Melhor paraTasks simples, bugs obvios
*comando --mode yolo
Pre-Flight

Planeja primeiro

O agente cria um plano detalhado, revisa com voce, e SO DEPOIS executa. Maximo de seguranca. Ideal para mudancas criticas onde o erro custa caro.
Perguntas10 — 15
Controle█████ Maximo
Melhor paraFeatures criticas
*comando --mode preflight

💡 Como usar os modos na pratica

Basta adicionar --mode yolo ou --mode preflight ao final de qualquer comando. Sem flag = modo interativo (padrao). Exemplos:

  • *create-prd — Cria o PRD de forma interativa (te faz perguntas)
  • *create-prd --mode yolo — Cria o PRD 100% autonomo, sem perguntar
  • *validate-story-draft --mode yolo — Valida e aprova a story automaticamente
  • *develop-yolo — Caso especial do dev: ja tem comando proprio (equivalente a *develop --mode yolo)
  • *review {story} --mode yolo — QA revisa sem pedir confirmacao
!

Coisas Importantes

Leia antes de usar. Serio.

🔒 So o /AIOX:agents:devops pode publicar codigo

Nenhum outro agente pode fazer git push, criar Pull Request ou gerenciar releases. Se voce pedir para o /AIOX:agents:dev fazer push, ele vai recusar e pedir para voce chamar o /AIOX:agents:devops. Isso existe para evitar que codigo nao revisado va para producao.

🔄 Se o QA reprovar, existe um loop automatico

Quando o /AIOX:agents:qa retorna FAIL, o AIOX ativa um ciclo:

  • /AIOX:agents:qa gera lista de correcoes com *create-fix-request
  • /AIOX:agents:dev aplica as correcoes com *apply-qa-fixes
  • /AIOX:agents:qa revisa de novo — repete ate 5 vezes
  • Se nao resolver em 5 tentativas, escala para o /AIOX:agents:aiox-master

🌱 Greenfield vs Brownfield

Voce vai ver esses termos em varios comandos. A diferenca e simples:

  • Greenfield = projeto NOVO, do zero. Nao existe codigo ainda. Use *create-prd, *create-full-stack-architecture.
  • Brownfield = projeto EXISTENTE. Ja tem codigo, banco, usuarios. Use *create-brownfield-prd, *create-brownfield-architecture.

💡 Comandos universais

Esses comandos funcionam em QUALQUER agente:

  • *help — Mostra todos os comandos disponiveis do agente ativo
  • *guide — Mostra instrucoes detalhadas de uso
  • *yolo — Alterna modo de permissao (ask / auto / explore)
  • *session-info — Mostra info da sessao atual
  • *exit — Sai do agente

💰 Cada agente tem sua area — nao invada

O AIOX foi projetado com separacao rigida de responsabilidades. Respeite as fronteiras:

  • Criar story → so /AIOX:agents:sm
  • Validar story → so /AIOX:agents:po
  • Criar epic → so /AIOX:agents:pm
  • Fazer push / PR → so /AIOX:agents:devops
  • Schema de banco → so /AIOX:agents:data-engineer
  • Resolver conflitos entre agentes → /AIOX:agents:aiox-master
